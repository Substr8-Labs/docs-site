# Architecture

## Overview

Substr8 is a trust protocol for AI agent execution.

```
Agent Framework (LangGraph, AutoGen, etc.)
           │
           ▼
    ┌─────────────────┐
    │  Instrumentation │  ← substr8-langgraph
    └─────────────────┘
           │
           ▼
    ┌─────────────────┐
    │    RunProof     │  ← Cryptographic receipt
    └─────────────────┘
           │
           ▼
    ┌─────────────────┐
    │  Verification   │  ← CLI / Web / API
    └─────────────────┘
```

---

## Core Primitives

### RunProof

A cryptographically verifiable receipt containing:

| Component | Purpose |
|-----------|---------|
| **Header** | Run metadata (agent, timestamps, status) |
| **Trace** | Hash-chained event sequence |
| **Commitments** | Merkle root + signature |
| **Outputs** | Execution results |

### Hash Chain

Events are chained:
```
H₁ = hash(event₁, null)
H₂ = hash(event₂, H₁)
H₃ = hash(event₃, H₂)
```

Any modification breaks the chain.

### Merkle Root

Single hash committing to all events:
```
        Root
       /    \
      H₁₂    H₃₄
     /  \   /  \
    H₁  H₂ H₃  H₄
```

### Ed25519 Signature

Proof hash signed with private key, verifiable with public key.

## Trust Protocol (Future)

Substr8 is evolving toward a full trust protocol with five primitives:

```
Discovery    →  Find trusted agents
    ↓
Intent       →  Declare what should happen
    ↓
Execution    →  Perform the work (RunProof)
    ↓
Acceptance   →  Confirm the result
    ↓
Attestation  →  Independent verification
```

| Primitive | Status | Description |
|-----------|--------|-------------|
| Execution | ✅ V1 | RunProof generation and verification |
| Intent | 🔜 v2.1 | Commit to task before execution |
| Acceptance | 📋 Planned | Approval events between agents |
| Attestation | 📋 Planned | Third-party verification |
| Discovery | 📋 Future | Trust-based agent discovery |

## Modules

### FDAA (File-Driven Agent Architecture)

Define agents as files with cryptographic identity.

```
my-agent/
├── agent.yaml       # Agent definition
├── skills/          # Capabilities
└── policy.yaml      # Access control
```

Hash of definition = agent identity.

### GAM (Git-Native Agent Memory)

Semantic memory with git-like provenance.

```bash
substr8 gam remember "User preference: dark mode"
substr8 gam recall "user preferences"
```

### ACC (Agent Capability Control)

Runtime capability enforcement.

```yaml
# policy.yaml
capabilities:
  - tool: web_search
    allowed: true
  - tool: send_email
    allowed: false
    reason: "Requires approval"
```

### DCT (Deterministic Capability Tokens)

Tamper-evident audit ledger for all actions.

### RIL (Runtime Integrity Layer)

Sandboxed execution with verification.

## Verification Flow

```
1. Load RunProof JSON
2. Validate schema
3. Verify hash chain (each event links to previous)
4. Recompute Merkle root
5. Verify signature against public key
6. Return result
```

All checks must pass for valid proof.
